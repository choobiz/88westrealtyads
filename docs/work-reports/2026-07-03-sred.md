## SR&ED Project 1: First-party, session-correlated A/B/C/D conversion measurement without a managed analytics dependency

*Mapped from: Task 6 in 2026-07-03.md*

### 1. Technological Uncertainty (Line 242)
**Existing knowledge base:** the LP is a statically-exported Next.js app on Vercel (no server runtime of its own) with variant assignment done in edge middleware (`proxy.ts`) and events pushed to a GTM (`GTM-MVD6JFMR`) → GA4 dataLayer. Standard practice would read conversion-by-variant from GA4.

**Technological uncertainty:** it could not be established from the codebase whether `experiment_variant` was a *queryable* GA4 dimension — that depends on GTM container config invisible to the code — nor whether GA4 Data API access could be obtained. It was therefore unknown whether a reliable per-variant conversion rate could be produced at all, and if built independently, how to correlate a *view* and a later *lead* to the same visitor and the same variant across a stateless serverless front end and a separate logging tier, without cookies/PII beyond the existing cohort cookie, and without a CORS preflight blocking a fire-and-forget beacon on page unload.

**Why standard practice was insufficient:** GA4/GTM introduces an unverifiable configuration dependency; a naive `fetch` beacon with `application/json` triggers a CORS preflight that is unreliable during page transitions; counting raw events double-counts multi-load sessions, corrupting the denominator of a conversion rate.

### 2. Systematic Investigation (Line 244)
**Hypothesis:** a dependency-free first-party tier that (a) tags each event with the *existing* `_lp_ab_cohort` variant + a per-session id, (b) transmits via `sendBeacon` with a CORS-safe `text/plain` body to avoid preflight, and (c) aggregates by *distinct session* would yield a correct, self-owned per-variant CVR.

**Work performed:**
- Verified the constraint empirically: the variant reaches only the dataLayer (GTM), not a code-visible GA4 dimension; no GA4/GHL API credential exists in repo or workspace → GA4 path indeterminate.
- Built a `node:http`-only service (no framework) on the VM behind nginx; chose append-only JSONL for crash-safe, lock-free writes at LP volume, stored outside the git tree to survive deploys.
- Designed session correlation: client mints a `sessionStorage` id, fires `view` once per session (guard flag) and `lead` on submit, both carrying variant+sid. Server aggregates **distinct sids** per variant for the denominator (sessions) and numerator (leads).
- Solved the beacon transport: `navigator.sendBeacon` with `Blob{type:"text/plain"}` — a CORS-safe request that skips preflight; the server JSON-parses the text body. Verified POST 204 end-to-end through nginx and a 401 gate on the stats/dashboard.
- Integrated with the running VM fleet: PM2 app, nginx `location` insertion into a live shared config (backup + `nginx -t` before reload to avoid taking down co-located services), token from a git-ignored `.env`.

**Evidence:** commits `79608a5`, `487d701`, `5552ce8`; `scripts/track-server.mjs`, `lib/track.ts`; smoke test (D 1 session / 1 lead / 100% CVR, then purged); bundle-string verification of the deployed client.

### 3. Technological Advancement (Line 246)
- A reusable pattern for **self-owned, session-correlated A/B conversion measurement** layered onto a stateless serverless front end + edge-assigned cohorts, independent of any managed-analytics container config.
- Established that `sendBeacon` + `text/plain` is the reliable preflight-free transport for cross-origin first-party events, and that **distinct-session aggregation** (not raw event counts) is required for a correct CVR denominator.
- Established a safe insertion procedure for adding a service to a live shared nginx vhost (backup → validate → reload) without collateral downtime.

**Residual uncertainty:** session identity is per-browser-session (sessionStorage), so cross-device/return visits are undercounted — an accepted approximation for relative variant comparison.

## SR&ED Project 2: Client-side geocoded list/map explorer over a heterogeneous, partially-identified listing set

*Mapped from: Tasks 3 + 5 in 2026-07-03.md*

### 1. Technological Uncertainty (Line 242)
**Existing knowledge base:** the listing pipeline emits two shapes — a "shown" set (mls stripped) and a "pending" set (mls retained) — with free-text addresses and no coordinates. Rendering an interactive map in a Next.js app that statically exports, without adding a heavy React map dependency, and merging the two heterogeneous sets without double-counting, were the open problems.

**Technological uncertainty:** it was unknown (a) how to key a de-duplication across two sets that do not share a stable identifier (one drops mls), while preserving genuinely-distinct same-building units; (b) how to obtain coordinates for free-text BC addresses at build time within rate limits; and (c) how to mount Leaflet in a statically-exported client component without SSR breakage and bridge raw-HTML map popups back into React handlers.

**Why standard practice was insufficient:** a mls-or-address dedup silently collapsed distinct units *and* double-counted cross-set listings (observed: 124 vs the correct 81); `react-leaflet` SSR-evaluates `window`; Leaflet popups are injected HTML with no React binding.

### 2. Systematic Investigation (Line 244)
**Hypothesis/Approach:** treat the shown set as source-of-truth, pre-filter pending by a normalized `street|area` key, and skip post-dedup (units are legitimately distinct); geocode via Nominatim with a persisted cache that retries nulls; load Leaflet from CDN inside `useEffect` and bridge popups through a `window.__fclPick(index)` global.

**Work performed:**
- Diagnosed the count inflation to the mls-vs-address key mismatch by inspecting both set shapes; corrected the merge; verified 81 unique.
- Built the geocoder with a 1.1s throttle + null-not-persisted cache (41/81 resolve; rest retry next build; all listings still list).
- Loaded Leaflet via unpkg in a client-only effect (no npm dep, no SSR eval); wired CartoDB tiles + circle markers + the popup→React bridge.
- Coupled with the curator ungate (Task 5): photo-less listings surface with a component-rendered branded placeholder, keeping the unblurred hero tier real-photo-only.

**Evidence:** commits `d8b7cce`, `abbfbb4`; `scripts/build-explorer-deals.mjs`, `components/foreclosure/ForeclosureListExplorer.tsx`, `scripts/curate-deals.mjs`.

### 3. Technological Advancement (Line 246)
- A merge/dedup rule for two-shape listing sets lacking a shared key that preserves distinct units while eliminating cross-set duplicates.
- A CDN-loaded, SSR-safe Leaflet integration + popup→React bridge pattern for statically-exported Next apps.
- A build-time free-text geocoding cache with null-retry semantics suitable for a daily cron.

**Residual uncertainty:** ~half of free-text addresses do not resolve on Nominatim; a higher-coverage geocoder (or Street View by resolved coords) is a follow-up.

## SR&ED Project 3: Automated photo sourcing for scraped listings + self-updating cron

*Mapped from: Task 4 in 2026-07-03.md*

### Summary (condensed)
**Uncertainty:** whether the pending-photo backlog was recoverable and could be prevented recurring within an unattended git-pull-then-execute cron. **Investigation:** ran the Tavily building-name image sourcer and established empirically that recovery is bounded by *named* buildings — nameless plain-address houses are unmatchable by this method (35 houses, 9 condos, 7 land remain); automated the sourcer into the cron and characterized the self-update hazard (PM2 runs the on-disk runner version at trigger time, so a pulled change lands one run later) and mitigated it. **Advancement:** a bound on automated photo coverage by listing-name availability, and a safe self-updating-cron pattern with an in-runner `.env` loader for secret delivery without committing keys.
